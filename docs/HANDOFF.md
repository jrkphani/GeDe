## How to work (non-negotiables)

1. **TDD, red first.** Each issue has a *Test-first plan* — write those tests, watch them fail, then implement. This has caught a real bug in every slice so far: a Zustand infinite-render loop from an unstable `?? []` selector fallback (003); a `flexRender` cell-identity remount bug and a load-vs-mutation race condition, both in the parameters/contexts stores (004, see below).
2. **`npm run verify`** (typecheck → eslint → **stylelint** → vitest → playwright) must be green before an issue is SHIPPED. A **husky pre-push** hook runs `verify:fast` (everything but e2e) and **GitHub Actions** (`.github/workflows/verify.yml`) runs the full `verify` on PRs; pre-commit runs `lint-staged`. Currently green: 183 unit/component (incl. a fast-check property test) + 17 e2e, typecheck + eslint(0/0) + stylelint clean.
3. **Schema changes only via `npm run db:generate`** (drizzle-kit) — never hand-write DDL. Migrations apply via the browser-safe runner in `src/db/migrate.ts` (`import.meta.glob` + `__migrations` ledger). Rename the generated file + update `meta/_journal.json`'s `tag` to match (project convention — `0000_init`, `0001_dimensions`, `0002_parameters`, `0003_contexts_bindings`); the runner globs by filename, not the journal, so this is cosmetic but expected.
4. **Layer boundaries are lint-enforced**: components must not import `src/db/**` (typescript-eslint no-restricted-imports). Writes: component → store action → `src/db/mutations.ts`. Keep it that way; extend the rule if you add layers.
   - **Also enforced (issue 020)**: no raw `<button>`/`<input>`/`<select>` and no direct `@radix-ui`/`cmdk` imports outside `src/components/ui/`; no hardcoded colors in CSS (stylelint). All of these fail `npm run verify`.
5. **Ship ritual**: set issue Status → SHIPPED, `git mv` it to `docs/issues/done/`, update the README index row (`done/` link + ✅), commit with the issue number, push. One commit per issue.
6. **References lines in issues are binding** — deviating from a cited SPEC/STYLE_GUIDE/SITEMAP section is a spec discussion, not an implementation choice.
7. **New UI composes the shared primitives** in `src/components/ui/` — `Button`, `InlineEdit`, `PhantomInput`, `Popover`/`PopoverContent`/`PopoverTrigger`, `Command`, `Swatch`/`SwatchButton`, `Input`. Don't hand-roll a button/input/popover; extend a primitive or add a new one to `ui/`. See ADR-0007. `EditableGrid` is the one grid primitive that owns its own cells (exempt from the raw-element rule).

## Architecture facts you'll need

- **DB**: PGlite (Postgres-in-WASM), `idb://gede` in browser, `memory://` in tests. `getDatabase()` in `src/db/client.ts` is a **singleton — never instantiate two PGlites on the same idb dir** (StrictMode double-mount deadlocked + crashed WASM; that's why the singleton exists).
- **Stores** (Zustand): `projects`, `dimensions`, `parameters`, `contexts`, `status` share one db handle via `src/store/database.ts` (set by `projects.init()`, or `setDatabase(db)` directly in tests). Reset helpers exist per store for tests.
- **Shell slots**: tiers put content in the context bar by rendering `<ContextBar>…</ContextBar>` (portal; band collapses when empty). All user feedback goes through `useStatusStore.announce(message, {label, run})` — no toasts, ever.
- **Routing**: `src/shell/routes.ts` (pure parse/serialize) + `src/shell/router.ts` (`useRoute`/`navigate`). Add route shapes to the type + both functions + the round-trip test.
- **Undo/redo** (`src/store/commandLog.ts`, issue 006): shared `useCommandLogStore` — every store action pushes one `{ label, undo, redo }`; `batch(label, fn)` groups several store calls from one user gesture (e.g. phantom-row create + first justification) into a single undo step. ⌘Z/⇧⌘Z live in `AppShell`, capture-phase (see the gotcha below). Session-scoped only — cleared whenever the open project changes, never persisted. Don't add per-feature undo buttons — reuse the status-bar action (`announce(message, { label, run })`) or the global shortcut; the projects-list archive flow's inline "Undo" now calls `useCommandLogStore.getState().undo()` directly instead of a bespoke per-store `undoLast`.
- **UI primitive layer** (`src/components/ui/`, issues 018–020, ADR-0007): shadcn-model primitives on **Tailwind v4** — but `tokens.css` stays the source of truth; `theme-bridge.css` maps shadcn semantic names onto GeDe tokens one-directionally. Primitives reuse the existing `.row-action`/`.inplace-input` classes (cascade-connected) rather than utility-restyling. Compose these; don't hand-roll (lint-enforced). Tailwind utilities are available for *new* styling.
- **EditableGrid** (`src/components/EditableGrid.tsx`, issue 004; `multiline` kind + `data-row-id` added issue 005): the shared TanStack-Table-backed grid for all tiers (ADR-0004). Column config is `{ id, header, cell: { kind: 'text'|'mono'|'combobox'|'static'|'multiline', ... } }`. Callers own `getValue`/`onCommit`; the grid owns editing/navigation (click/Enter edits, Enter commits + moves down a row, Esc reverts, arrow keys navigate cells when not editing, native Tab/Shift-Tab traverses since every cell is a real focusable element in DOM order). `multiline` is a `<textarea>` that auto-grows while editing and clamps to 2 lines on display (`title` carries the full text) — the one sanctioned row-height exception (STYLE_GUIDE §6); Enter still commits (no newline-on-Enter), Shift+Enter inserts one. Every `<tr>` carries `data-row-id={getRowId(row)}` — a generic hook for external "focus this row" navigation (used by the issue 005 duplicate badge; no canvas yet). Tier 1/2 tables (013/014) should reuse this component unchanged per its acceptance criteria. **Never inline the `cell` function per-column-per-render** — see the flexRender gotcha below.
- **ContextRegister** (`src/components/ContextRegister.tsx`, issue 004): dynamic columns from live dimensions (Symbol · one per dimension · Justification · Children — Children is a placeholder until issue 011). Draft vs complete is computed live via `isComplete()`, not stored. Combobox options come from `useParametersStore`; symbol edits go through `setContextSymbol` (rejects collisions, announced via the status bar).
- **Canvas** (`src/components/Canvas.tsx` + `src/domain/canvasLayout.ts`, issue 008; read-only — selection/spokes arrive issue 009): the layout module is pure (`layout(input) → geometry`, zero React/store imports, SPEC invariant 5) — a fixed 1000×1000 abstract space, `d3-shape` arc paths, context-node position = centroid of bound parameter dots + hash-seeded jitter, `d3-force` collision run synchronously for a fixed tick count. `ArcGeometry.d` is centered at `(0,0)` (a `d3.arc()` quirk, it has no center option) — always render it inside `transform="translate(${CENTER},${CENTER})"` (`CENTER` is exported from `canvasLayout.ts` specifically so this can't drift). Renders beside `ContextRegister` inside `DesignSurface.tsx`'s `view === 'canvas'` branch (SPEC invariant 6), reading the same dimensions/parameters/contexts/bindings state via its own independent store subscriptions.

## Gotchas already paid for (don't rediscover)

- **RTL auto-cleanup doesn't run** without vitest globals — `src/test/setup.ts` registers it manually. Component tests need `// @vitest-environment jsdom` as line 1.
- **Radix popovers close on Escape at document level** — in-place editors inside popovers need the `onEscapeKeyDown` guard (see DimensionManager) to honor the SITEMAP §4 Esc order.
- **Never publish related store updates in two `set()` calls** if a surface derives visibility from the first — the guided-start swap unmounted an open editor mid-gesture until `add()` set `dimensions` + `editingId` atomically. Pattern: one gesture = one atomic `set()`.
- **Selector fallbacks must be stable references** — `s.byX[id] ?? []` creates a new array every call and loops Zustand's subscription forever; use a module-level `const EMPTY: T[] = []` instead (bit us in issue 003's `ParameterList`).
- **A `load()`-style hydration action can lose a race against a mutation for the same key, even when `load()` was called after the mutation started.** `src/store/parameters.ts`'s `add()` does a SELECT-then-INSERT internally; a concurrent `load()` (fired by a per-dimension mount effect) can slot its own read *between* those two steps and read pre-write state, then commit that stale empty result over the mutation's fresh one. Fixed with a per-key `generation` counter: every mutation bumps it synchronously before awaiting anything; `load()` snapshots the generation before its read and discards its result if the generation moved — mutations always win regardless of DB-level interleaving. If a store has more than one independent caller of `load()` for the same key (as parameters does: each `ParameterList` *and* `ContextRegister` both load per-dimension), assume this race is possible and apply the same pattern.
- **`flexRender(columnDef.cell, ctx)` treats the `cell` function as a component *type*.** If you rebuild the columns array (and thus each column's `cell` closure) fresh on every parent render — natural when `getValue`/`onCommit` close over live store state — React remounts the cell subtree on *every* re-render, including the one triggered by opening a popover itself, silently resetting any local state (like a combobox's `open`) the instant it's set. Fix: one **stable, module-level** `cell` function for every column (`renderGridCell` in `EditableGrid.tsx`), with per-column config + the live `nav` object passed through TanStack's `meta` option (read at render time via `info.table.options.meta`, never captured in a closure).
- **jsdom is missing `Element.prototype.hasPointerCapture`/`setPointerCapture`/`releasePointerCapture`, `scrollIntoView`, and `window.ResizeObserver`.** Radix Popover's trigger and cmdk's list-scroll effect call these; without polyfills, Radix silently fails to open (no error surfaced) and cmdk throws inside a layout effect that aborts the in-flight commit. Polyfilled as no-ops in `src/test/setup.ts`, guarded with `typeof Element !== 'undefined'` / `typeof window !== 'undefined'` since the plain `src/db/*.test.ts` files run in the `node` environment, not jsdom.
- **e2e**: two "Undo"-named buttons exist (app bar disabled + status bar) — scope selectors (`page.locator('.status-bar')…`). After clicking "Add dimension", wait for the row editor before sending keys. Once the context register exists on the same page, dimension names ("Value", "Stake", …) collide with register column headers in plain `page.getByText(...)` locators — scope to `.dim-manager`/`.dim-row__name` explicitly (bit both `parameters.spec.ts` and `context-register.spec.ts`). After a phantom-row creation or a store mutation, wait for its visible effect (e.g. `toBeVisible()`/`toContainText()`) before the next keyboard action — the phantom clears and refocuses optimistically but the underlying store write is still in flight.
- **PGlite must stay in `optimizeDeps.exclude`** (vite.config.ts) or its WASM breaks in dev.
- **A bubble-phase `window` keydown listener can be silently swallowed by a descendant's `stopPropagation()`.** `EditableGrid`'s phantom-row input calls `e.stopPropagation()` unconditionally (to keep its own Enter/Escape/arrow grammar self-contained) — since committing a cell edit moves focus to the *next* row (often the phantom one, per the Numbers grammar), any global shortcut registered as `window.addEventListener('keydown', fn)` never fires right after a commit. Fixed for ⌘Z/⇧⌘Z by registering on the **capture phase** (`addEventListener(..., true)`, matching `removeEventListener`); the pre-existing ⌘1/2/3 tier-switch handler in the same file has the same latent bug but wasn't touched (out of scope for issue 006). If you add another global shortcut, register it capture-phase too, or you'll rediscover this the hard way — it only surfaced via the e2e test, not unit tests, because jsdom component tests don't naturally land focus on a phantom input mid-flow the way a real browser session does.
- **Deferring a global shortcut to "native" behavior (e.g. text-field undo) must check the field actually has content**, not just that it's focused. An unconditional "skip while any input/textarea is focused" guard for ⌘Z looks right until the input is *empty* (e.g. the phantom row right after a commit) — then it silently eats the keystroke for nothing, since there's no native undo history in an empty field. Check `el.value.length > 0` (or `isContentEditable`) before deferring.

- **A store mutation that cascades into another store's state doesn't need bespoke diffing — re-read the affected keys from the DB.** `dimensionsStore.remove()` (issue 007) hard-deletes bindings across contexts it doesn't own; rather than hand-rolling an in-memory patch of `useContextsStore`'s `bindingsByContext`, it calls a new `syncBindingsForContexts(contextIds)` action that just re-fetches those contexts' bindings from the DB (the mutation layer already wrote the authoritative rows). Simpler than diffing, and safe here specifically because it runs *synchronously after* the write it's reacting to — not a concurrent `load()`, so the generation-counter race (below) doesn't apply.
- **A `load(id)`-style action must set the id/scope field synchronously, before its own DB round-trip — not just the data that round-trip returns.** `useContextsStore.load(projectId)` only called `set({ projectId, ... })` at the very end, after awaiting `listContexts`/`fetchBindingsMap`. `create()`/`setSymbol()`/etc. read `get().projectId` internally rather than taking it as an argument; `ContextRegister`'s mount effect fires `void loadContexts(projectId)` without awaiting it, and the register (phantom row included) is interactive on that same render. A context created fast enough after mount — before `load()`'s DB round-trip resolved — saw `projectId` still `null` and silently no-op'd: no error, no context, the phantom just optimistically cleared forever. **100% reproducible on GitHub Actions' CI runner, 0% locally** — the CI box was consistently slow enough to lose the race every single run (traced back through the previous two commits, discovered while chasing what looked at first like ordinary e2e flakiness); a fast local machine never lost it. Root-caused by adding temporary `console.log` instrumentation to the actual code path and reading a downloaded Playwright trace's console output on a real failing CI run — guessing from timeouts/retries alone was a dead end (see the two failed attempts before this one in git log around 2026-07-05). Fix: `set({ projectId })` as the very first line of `load()`, since the caller already knows it synchronously. If you add a new per-scope `load()`, apply this alongside the generation-counter pattern above, not instead of it — they guard different halves of the same class of bug (the scope key vs. the data it loads).
- **When an e2e failure won't reproduce locally under any configuration (cold server, forced worker concurrency, longer timeouts), don't keep guessing — add `trace: 'retain-on-failure'` / `screenshot: 'only-on-failure'` to `playwright.config.ts` and an `actions/upload-artifact` step to CI (`if: failure()`), then `gh run download <id> -n playwright-report` and read the trace's `frame-snapshot`/`console`/`before`/`after` events directly (they're JSONL inside the downloaded zip — no need for `npx playwright show-trace`'s UI). This is how both the stray-popover-click bug and the `projectId` race above were actually found, after two rounds of plausible-sounding-but-wrong fixes (a timeout bump, a stale test comment) didn't move the needle.

- **`d3.arc()()` always emits path data centered at `(0,0)` — it has no "center" config option**, unlike every other geometry helper in `canvasLayout.ts` (`pointAt()`) which bakes the `(CENTER,CENTER)` offset in directly. Forgetting to wrap the arc `<path>` in a `transform="translate(...)"` renders a barely-visible sliver near the SVG's top-left corner instead of a circle — looked like a totally broken layout at first glance, but unit/component tests (which only assert path-string non-emptiness and coordinate finiteness, not visual position) never caught it. Only a real rendered screenshot did. If you add another d3-shape generator to the canvas, assume the same "centered at origin" default and check it visually, not just via `d.length > 0`.
- **d3-force's collision resolution falls back to `Math.random()` (an internal `jiggle()`) only when two nodes start at the *exact same* coordinate** — a real risk here since a context with zero bindings, or two contexts on an identical tuple, would otherwise coincide exactly. ADR-0005 forbids any randomness. Fixed per SPEC §4.2's own wording ("hash-seeded jitter for collisions"): seed every node's initial position with a deterministic hash of its own (stable) context id before handing it to the simulation, so distinct contexts never coincide and the random path is never exercised. If you add more force-directed geometry, apply the same pattern rather than trusting d3-force's defaults to "just be deterministic" — they aren't, in this one specific case.
- **A CSS `@container` query cannot target the same element its `container-type` is set on** — only descendants can respond to a container's size. `container-type: inline-size` on `.design-surface-row` plus `@container (max-width: 640px) { .design-surface-row { ... } }` silently never matched (no error, just never applied); moving `container-type` to the parent (`.design-main`) fixed it. Easy to miss because nothing errors — the query just quietly never fires.
- **Reusing an existing CSS class for a new, differently-shaped layout can silently wreck it.** `DesignSurface`'s new canvas+register row reused the `.projects` class (`max-width: 560px`), which existed for the single centered panel of the projects list and had never mattered for the register-only view before. The result rendered — no error, no failing test — just squeezed into a ~510px box. `npx tsc`/eslint/stylelint/vitest all stayed green throughout; only a manual browser screenshot caught it. When a new feature shares a wrapper element with an existing one, check what CSS classes that wrapper already carries and whether they were designed for the old shape.
- **Manual browser verification is not optional for new visual/layout work, even with full unit+component+e2e coverage.** All three of the bugs above (arc centering, panel-width squeeze, self-querying container) passed `npm run verify` clean — the component tests asserted the right *data* (arc count, path non-emptiness, node count) but none of them could catch wrong *geometry* or wrong *CSS layout*, since jsdom doesn't lay out real pixels. Driving the actual dev server with Playwright (`chromium` via the already-installed `playwright-core`, since only `@playwright/test` is a direct dependency — `import { chromium } from '.../node_modules/playwright-core/index.mjs'`) and looking at real screenshots is what found all three.

Root cause turned out to be **not** a layout bug — `getBoundingClientRect` proved the column boundaries are geometrically correct; the register-scroll container was genuinely scrolled (a native browser scroll-into-view fires whenever the phantom row's Justification input is focused off-screen, which is common once issue 005's two extra columns push the table past typical container widths). A frozen `left:0` sticky column legitimately occludes whatever scrolls behind it — that's the feature, not a bug — but there was no visual depth cue, so it read as broken/clipped text. Real fix: `.editable-grid` was `border-collapse: collapse`, which silently suppresses `box-shadow` on table cells in every browser; switched to `border-collapse: separate; border-spacing: 0` (renders identically here since no two adjacent cells declare the same shared border edge) and added a `--shadow-frozen-col` token so the frozen column now casts a visible depth cue, making the layering read as intentional. **Do not try to fix this class of issue with column `width`/`padding` tweaks** — the auto-scroll distance is derived from the table's own total width, so widening any column upstream of the scroll target proportionally *increases* the scroll distance and can make the overlap worse (verified empirically, not just theorized).
