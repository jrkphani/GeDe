## How to work (non-negotiables)

1. **TDD, red first.** Each issue has a *Test-first plan* — write those tests, watch them fail, then implement. This has caught a real bug in every slice so far: a Zustand infinite-render loop from an unstable `?? []` selector fallback (003); a `flexRender` cell-identity remount bug and a load-vs-mutation race condition, both in the parameters/contexts stores (004, see below).
2. **`npm run verify`** (typecheck → eslint → **stylelint** → vitest → playwright) must be green before an issue is SHIPPED. A **husky pre-push** hook runs `verify:fast` (everything but e2e) and **GitHub Actions** (`.github/workflows/verify.yml`) runs the full `verify` on PRs; pre-commit runs `lint-staged`. Currently green: 90 unit/component + 11 e2e, typecheck + eslint(0/0) + stylelint clean.
3. **Schema changes only via `npm run db:generate`** (drizzle-kit) — never hand-write DDL. Migrations apply via the browser-safe runner in `src/db/migrate.ts` (`import.meta.glob` + `__migrations` ledger). Rename the generated file + update `meta/_journal.json`'s `tag` to match (project convention — `0000_init`, `0001_dimensions`, `0002_parameters`, `0003_contexts_bindings`); the runner globs by filename, not the journal, so this is cosmetic but expected.
4. **Layer boundaries are lint-enforced**: components must not import `src/db/**` (typescript-eslint no-restricted-imports). Writes: component → store action → `src/db/mutations.ts`. Keep it that way; extend the rule if you add layers.
   - **Also enforced (issue 020)**: no raw `<button>`/`<input>`/`<select>` and no direct `@radix-ui`/`cmdk` imports outside `src/components/ui/`; no hardcoded colors in CSS (stylelint). All of these fail `npm run verify`.
5. **Ship ritual**: set issue Status → SHIPPED, `git mv` it to `docs/issues/done/`, update the README index row (`done/` link + ✅), commit with the issue number, push. One commit per issue.
6. **References lines in issues are binding** — deviating from a cited SPEC/STYLE_GUIDE/SITEMAP section is a spec discussion, not an implementation choice.
7. **New UI composes the shared primitives** in `src/components/ui/` — `Button`, `InlineEdit`, `PhantomInput`, `Popover`/`PopoverContent`/`PopoverTrigger`, `Command`, `Swatch`/`SwatchButton`, `Input`. Don't hand-roll a button/input/popover; extend a primitive or add a new one to `ui/`. See ADR-0007. `EditableGrid` is the one grid primitive that owns its own cells (exempt from the raw-element rule).

## Architecture facts you'll need

- **DB**: PGlite (Postgres-in-WASM), `idb://gede` in browser, `memory://` in tests. `getDatabase()` in `src/db/client.ts` is a **singleton — never instantiate two PGlites on the same idb dir** (StrictMode double-mount deadlocked + crashed WASM; that's why the singleton exists).
- **Stores** (Zustand): `projects`, `dimensions`, `parameters`, `contexts`, `status` share one db handle via `src/store/database.ts` (set by `projects.init()`, or `setDatabase(db)` directly in tests). Reset helpers exist per store for tests.
- **Shell slots**: tiers put content in the context bar by rendering `<ContextBar>…</ContextBar>` (portal; band collapses when empty). All user feedback goes through `useStatusStore.announce(message, {label, run})` — no toasts, ever.
- **Routing**: `src/shell/routes.ts` (pure parse/serialize) + `src/shell/router.ts` (`useRoute`/`navigate`). Add route shapes to the type + both functions + the round-trip test.
- **Undo**: currently single-step inverse per gesture (archive only). Issue 006 replaces this with a command log; every store action is already "one gesture = one call" to be that seam. Don't add per-feature undo buttons — reuse the status-bar action.
- **UI primitive layer** (`src/components/ui/`, issues 018–020, ADR-0007): shadcn-model primitives on **Tailwind v4** — but `tokens.css` stays the source of truth; `theme-bridge.css` maps shadcn semantic names onto GeDe tokens one-directionally. Primitives reuse the existing `.row-action`/`.inplace-input` classes (cascade-connected) rather than utility-restyling. Compose these; don't hand-roll (lint-enforced). Tailwind utilities are available for *new* styling.
- **EditableGrid** (`src/components/EditableGrid.tsx`, issue 004): the shared TanStack-Table-backed grid for all tiers (ADR-0004). Column config is `{ id, header, cell: { kind: 'text'|'mono'|'combobox'|'static', ... } }`. Callers own `getValue`/`onCommit`; the grid owns editing/navigation (click/Enter edits, Enter commits + moves down a row, Esc reverts, arrow keys navigate cells when not editing, native Tab/Shift-Tab traverses since every cell is a real focusable element in DOM order). Tier 1/2 tables (013/014) should reuse this component unchanged per its acceptance criteria. **Never inline the `cell` function per-column-per-render** — see the flexRender gotcha below.
- **ContextRegister** (`src/components/ContextRegister.tsx`, issue 004): dynamic columns from live dimensions (Symbol · one per dimension · Justification · Children — Children is a placeholder until issue 011). Draft vs complete is computed live via `isComplete()`, not stored. Combobox options come from `useParametersStore`; symbol edits go through `setContextSymbol` (rejects collisions, announced via the status bar).

## Gotchas already paid for (don't rediscover)

- **RTL auto-cleanup doesn't run** without vitest globals — `src/test/setup.ts` registers it manually. Component tests need `// @vitest-environment jsdom` as line 1.
- **Radix popovers close on Escape at document level** — in-place editors inside popovers need the `onEscapeKeyDown` guard (see DimensionManager) to honor the SITEMAP §4 Esc order.
- **Never publish related store updates in two `set()` calls** if a surface derives visibility from the first — the guided-start swap unmounted an open editor mid-gesture until `add()` set `dimensions` + `editingId` atomically. Pattern: one gesture = one atomic `set()`.
- **Selector fallbacks must be stable references** — `s.byX[id] ?? []` creates a new array every call and loops Zustand's subscription forever; use a module-level `const EMPTY: T[] = []` instead (bit us in issue 003's `ParameterList`).
- **A `load()`-style hydration action can lose a race against a mutation for the same key, even when `load()` was called after the mutation started.** `src/store/parameters.ts`'s `add()` does a SELECT-then-INSERT internally; a concurrent `load()` (fired by a per-dimension mount effect) can slot its own read *between* those two steps and read pre-write state, then commit that stale empty result over the mutation's fresh one. Fixed with a per-key `generation` counter: every mutation bumps it synchronously before awaiting anything; `load()` snapshots the generation before its read and discards its result if the generation moved — mutations always win regardless of DB-level interleaving. If a store has more than one independent caller of `load()` for the same key (as parameters does: each `ParameterList` *and* `ContextRegister` both load per-dimension), assume this race is possible and apply the same pattern.
- **`flexRender(columnDef.cell, ctx)` treats the `cell` function as a component *type*.** If you rebuild the columns array (and thus each column's `cell` closure) fresh on every parent render — natural when `getValue`/`onCommit` close over live store state — React remounts the cell subtree on *every* re-render, including the one triggered by opening a popover itself, silently resetting any local state (like a combobox's `open`) the instant it's set. Fix: one **stable, module-level** `cell` function for every column (`renderGridCell` in `EditableGrid.tsx`), with per-column config + the live `nav` object passed through TanStack's `meta` option (read at render time via `info.table.options.meta`, never captured in a closure).
- **jsdom is missing `Element.prototype.hasPointerCapture`/`setPointerCapture`/`releasePointerCapture`, `scrollIntoView`, and `window.ResizeObserver`.** Radix Popover's trigger and cmdk's list-scroll effect call these; without polyfills, Radix silently fails to open (no error surfaced) and cmdk throws inside a layout effect that aborts the in-flight commit. Polyfilled as no-ops in `src/test/setup.ts`, guarded with `typeof Element !== 'undefined'` / `typeof window !== 'undefined'` since the plain `src/db/*.test.ts` files run in the `node` environment, not jsdom.
- **e2e**: two "Undo"-named buttons exist (app bar disabled + status bar) — scope selectors (`page.locator('.status-bar')…`). After clicking "Add dimension", wait for the row editor before sending keys. Once the context register exists on the same page, dimension names ("Value", "Stake", …) collide with register column headers in plain `page.getByText(...)` locators — scope to `.dim-manager`/`.dim-row__name` explicitly (bit both `parameters.spec.ts` and `context-register.spec.ts`). After a phantom-row creation or a store mutation, wait for its visible effect (e.g. `toBeVisible()`/`toContainText()`) before the next keyboard action — the phantom clears and refocuses optimistically but the underlying store write is still in flight.
- **PGlite must stay in `optimizeDeps.exclude`** (vite.config.ts) or its WASM breaks in dev.
