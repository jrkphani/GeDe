## Patterns (this run — reuse)

### Store-factory invariants (100 Phase A — CRITICAL for Phase B)
- **The circular-init safety is an IMPLICIT invariant, unlinted.** `canvasStores.ts` ↔ the 3 store modules form an import cycle; it is safe ONLY because (1) `createContextsStore`/`createDimensionsStore`/`createCanvasComposeStore` are **hoisted `function` declarations** (callable during module init in any entry order) and (2) the dependent modules import `CanvasStores` as a **type-only import** (erased at compile time). A ⚠️ comment atop `canvasStores.ts` pins this. **Do NOT** convert a factory to a `const` arrow export, and **do NOT** add a value import from `canvasStores.ts` into contexts/dimensions/canvasCompose — either silently reintroduces a TDZ/undefined-binding crash with no lint/compiler signal (only a runtime crash in specific import orders). The 1665-test suite + prod build are the only guardrails.
- **Sibling reads go through `getStores().use*`, never the module singleton** (dimensions→contexts ×3; compose→contexts ×9, →dimensions ×4). `paramNameById` KEEPS the global `useParametersStore` (parameters isn't split). A leftover direct-singleton reference is correct only for the default instance and a latent Phase-B bug — grep for it when adding instances.
- **`getCanvasStores(non-null)`/`releaseCanvasStores` are dead code until Phase B** → their registry/teardown/multi-instance resolution has ZERO existing coverage. Phase B/D must add it. `stores.teardown` hand-wires contexts+dimensions only (compose has no sub today).
- **Undo stays ONE GLOBAL history** (owner decision): all instances share `commandLog`/`status`/`sync`. Undo closures capture the instance's `set`/`get`, so entries bind correctly; Phase B+ must reckon with cross-canvas undo interleaving in the single LIFO.
