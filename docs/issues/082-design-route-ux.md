# 082: Design route (tier 2) ‚Äî stable, consistent, keyboard-fast dimension/parameter entry + on-ring authoring

- **Status**: OPEN
- **Milestone**: M6 (UI polish, same track as 013/021/024/027/081) ‚Äî no sync/infra blocker. Design-route surgery only; no schema, no synced-column ripple (unlike 081), so it is a pure client/render change.
- **Blocked by**: none hard. **Sequence after 081 lands** ‚Äî 081 edits Foundation (tier 1) source; this issue edits Design (tier 2) source; different files, no merge conflict, but running both red at once fights `verify:fast` for no reason. Phase 2 (on-ring editing) additionally wants Phase 1's stable canvas box in place first.

## User story

As a designer building the 2nd Tier ‚Äî Design, I want to **enter as many dimensions and parameters as I like in the shortest possible time, without the circular visual jumping out from under me**. Today three things fight that: (1) the "Add dimension" affordance physically relocates the instant the circle becomes drawable; (2) dimensions, parameters, and contexts are each added with a *different* grammar, and dimensions ‚Äî the most important object ‚Äî are the most expensive one; (3) the dimension/parameter editor uses a weaker keyboard model than the context register sitting inches to its right, so rapid typed entry breaks at every row boundary. I want one stable, consistent, keyboard-first entry path ‚Äî **and** I want to be able to draft directly on the ring itself when that's more natural.

## The four settled decisions (build the spec around these)

1. **Circle role = Hybrid.** The ring is a first-class add/edit surface **and** a table/rail remains the fast keyboard bulk-entry path. Neither is subordinate to the other.
2. **Layout = On-ring inline (Direction C)** ‚Äî ghost "+" ring-gap segments; click a gap to add a dimension arc; inline rename on the ring ‚Äî **combined** with the Hybrid table/rail path for fast typed bulk entry.
3. **n=2 floor = Soft hint.** Delete the `guided`/populated tree bifurcation entirely (`DesignSurface.tsx:364-411`). The circle renders partial/empty from the very first dimension; **nothing rebuilds at n=2**.
4. **Target = Desktop + tablet.** Both input paths need a deliberate narrow-width (<~640px) story; on-ring text editing and touch hit targets are designed for tablet from the start, not retrofitted.

## Investigation summary (grounds the design below ‚Äî real file:line from the UX audit)

- **The reflow is a whole-surface re-render, not a CSS nudge.** `DesignSurface` renders two entirely different trees gated on `guided`: `DesignSurface.tsx:366` sets `guided = dimensions.length < 2`; the guided branch (`:382-391`) makes the surface **be** `<DimensionManagerPanel />` inline in a `.panel`, so "Add dimension" is a big center-stage button; the populated branch (`:395-556`) collapses the manager into a **closed** Radix popover behind a `command`-variant trigger literally labeled **"Dimensions"** (`DimensionManager.tsx:250-253`), mounted in the top context bar (`DesignSurface.tsx:411`). The flip fires at `:364-368` the moment `dimensions.length >= 2 && editingId === null`. So the button is bound neither to the graph geometry nor to stable page chrome ‚Äî it is bound to a state bifurcation that fires exactly when "the circular graph is generated." The user's mental model is precisely correct. Decision 3 removes this bifurcation outright.
- **Secondary drift inside the popover.** The "Add dimension" button is the last child *after* the mapped dimension list (`DimensionManager.tsx:212-237`), so each added dimension pushes it further down and the popover scrolls it away.
- **Three different add grammars coexist within inches of each other:**

  | Object | Add affordance | Grammar | File:line |
  | --- | --- | --- | --- |
  | Context | Phantom row in the grid | Type + Enter; **Tab-across into a fresh row**; refocuses | `ContextRegister.tsx:266-278`, `EditableGrid.tsx:565-614` |
  | Parameter | Phantom input under each dimension | Type + Enter ‚Üí **refocuses itself** for the next | `ParameterList.tsx:86-92`, `inline-editor.tsx:148-156` |
  | **Dimension** | **"Add dimension" command button** | Click ‚Üí spawns a row in edit mode ‚Üí commit ‚Üí **dead-stops** | `DimensionManager.tsx:227-237`, store `dimensions.ts:85-111` (`editingId: row.id` at `:95`) |

  STYLE_GUIDE ¬ß6 states the canonical rule outright: *"New row = start typing in the phantom row."* Contexts and parameters obey it; dimensions do not (¬ß2.2 even sanctions the command button, so the inconsistency is partly codified ‚Äî but it is still friction against the north star). Decision-aligned fix: unify all three onto the phantom-row pattern.
- **The dense-typing surface has the weaker keyboard model.** The context register (`EditableGrid`) has the strong grammar: click-or-Enter to edit, Enter commits + moves **down** (`EditableGrid.tsx:300-304`), Tab commits + moves **right** (`:305-307`), Tab-from-phantom **creates a row and continues into it** (`:594-608`, `:714-749`), Esc reverts (`:308-313`), arrow-key cell nav (`:214-231`), autosave-on-commit. The dimension/parameter rows use the lighter `InlineEdit` (`inline-editor.tsx:99-108`): Enter commits then `setEditing(false)` ‚Äî **no advance**; **Tab falls through to native focus**, landing on the swatch/Remove button, not the next name; no arrow nav (only Alt+Arrow *reorder*, `DimensionManager.tsx:140-147`). So the surface where the user types the most (N dimensions √ó M parameters) is the one that can't be driven from the keyboard.
- **Keyboard-only rapid-entry seams (audit trace "add 5 dims + params, no mouse"):** (a) in populated mode the manager is a closed popover with **no** keyboard shortcut to open it, while `c`=New context and `v`=toggle view *are* bound (`DesignSurface.tsx:248-311`, SITEMAP ¬ß4) ‚Äî mouse required just to start; (b) dimension‚Üídimension advance broken (above); (c) no keyboard bridge dimension‚Üíits parameters; (d) param‚Üíparam works (`PhantomInput` refocuses, `inline-editor.tsx:151-152`); (e) crossing the n=2 floor mid-flow swaps the whole tree (`:364-368`) and focus is lost ‚Äî Decision 3 dissolves this seam entirely.
- **Adding a parameter reflows every sibling dot on that arc.** `canvasLayout.ts:278` ‚Äî `const slot = segmentSpan / (params.length + 1)`, dots placed at `startAngle + slot * (j + 1)` (`:279-281`). Arc span is fixed; params are evenly distributed, so adding param #4 re-angles all four, and every spoke bound to those dots (`Canvas.tsx:315-332`) moves with them ‚Äî object permanence broken during the exact act of "entering as much as you like." Contrast node migration, which is done *right*: a ~120ms reduced-motion-safe ease to the recomputed centroid via a CSS transform transition (`Canvas.tsx:355-362`). That transition is the template; the dot placement is the bug.
- **Layout anchors that already exist and should be reused:** `.canvas-toolbar` (`DesignSurface.tsx:471-480`, `base.css:1355-1359`) already stably hosts "New context" above the canvas row ‚Äî a proven always-anchored chrome slot on this exact route. `.design-surface-row` is a 40/60 canvas/register flex that collapses to a column <640px (`base.css:1295-1339`) ‚Äî the tablet fallback seam to design against. The invisible ‚â•44px hit circle is already sized from measured width by `dotHitRadiusUnits` (`canvasResponsive.ts:21-23`), which Phase 2 reuses for touch targets.

## Design brief

The north star is the yardstick for every decision: **get the user to enter as much information as they like in the shortest time, while keeping the circular visual stable.** Two input paths, neither subordinate (Decision 1):

- **Path A ‚Äî the fast typed rail (Phase 1).** A persistent dimension/parameter editor that (i) never unmounts across the n=2 floor and (ii) inherits the context register's real keyboard grammar. It is the fastest way to pour in ten dimensions and their parameters without touching the mouse.
- **Path B ‚Äî on-ring authoring (Phase 2).** Ghost "+" segments in the ring gaps; click a gap to add a dimension arc; inline-rename on the ring; click near an arc to add a parameter dot; inline-rename it. Direct manipulation for the user who thinks spatially ‚Äî the affordance *is* the circle, so it structurally cannot "move the button."

### Phase 1 brief ‚Äî soft-hint floor + persistent, keyboard-real rail

- **Delete the bifurcation (Decision 3).** Remove the `guided` state machine and the two-tree render (`DesignSurface.tsx:59` state, `:364-368` flip effect, `:378` gate, `:382-391` guided branch). The surface renders **one** tree at all dimension counts. The circle renders partial/empty from the first dimension (Canvas already handles 0/1-dimension geometry ‚Äî `canvasLayout.ts:243` returns empty arcs/dots/nodes for 0 dims, and draws a single arc for 1; its own empty prompt "Bind your first context" at `Canvas.tsx:415-426` stays as the soft hint). The n<2 state becomes a **hint**, not a gate: a quiet inline line ("Add a second dimension to start binding contexts") replaces the hard `placeholder` wall at `DesignSurface.tsx:386`, and the rail + circle are both live the entire time.
- **Persistent rail, always anchored.** The dimension/parameter editor becomes a stable region on the Design surface (not a popover that appears/disappears). Placement: a left rail inside `.design-surface-row` (rail ¬∑ canvas ¬∑ register), or ‚Äî the lower-risk first cut ‚Äî promote `DimensionManagerPanel` into an always-open panel in the same stable slot as `.canvas-toolbar` (`DesignSurface.tsx:471-480`), so it sits in fixed page chrome above/beside the canvas and **never** relocates. The context-bar "Dimensions" popover trigger (`DimensionManager.tsx:242-260`) is retired on the Design surface (the manager is no longer hidden), removing seam (a). Keep the `DimensionManager` popover export only if another surface still needs it; grep before deleting (`DimensionManager` / `DimensionManagerPanel` referenced in `DesignSurface.tsx:27,411,388`).
- **Unify the three add grammars onto the phantom-row pattern (Decision toward north star).** Replace the "Add dimension" command button (`DimensionManager.tsx:227-237`) with a phantom-row "type to add a dimension" input using the same primitive parameters already use (`PhantomInput`, `inline-editor.tsx:125-161`) ‚Äî Enter commits the name (color assigned from `DIMENSION_PALETTE` rotation as `add()` already does), clears, and **refocuses** for the next dimension. Parameters already use this (`ParameterList.tsx:86-92`) ‚Äî leave that grammar, extend its reach (below). Contexts already use it (`ContextRegister`/`EditableGrid`). Result: one add grammar for all three objects.
- **Raise the editor to the register's keyboard contract.** The dimension name and parameter name editors must gain: Enter-commits-and-advances-**down** to the next name; Tab-commits-and-moves-**right**/Shift-Tab-left across fields; Tab-from-the-phantom **creates and continues into the new row**; Esc reverts; arrow-key row nav. Two implementation routes ‚Äî pick per the build note below:
  - **T1 (smaller):** teach `InlineEdit` (`inline-editor.tsx`) an advance protocol mirroring `EditableGrid`'s `commitAndAdvance` (`EditableGrid.tsx:267-273`) + a hover/focus underline so "click to rename" is discoverable. Keeps the popover-free rail lightweight; con ‚Äî a second grammar to keep in sync with `EditableGrid`.
  - **T2 (unifying):** render the dimension list *as* an `EditableGrid` (columns: swatch ¬∑ name ¬∑ param-count ¬∑ remove; phantom row = "type to add"), inheriting the full grammar for free and making the two surfaces byte-identical. Parameters stay nested (drill-in or a nested phantom, since they don't map to grid rows cleanly). Con ‚Äî a grid inside a rail is heavier; swatch needs a new cell kind.
  - **Recommendation:** T2 for the north-star payoff (one grammar, one code path) if the rail can host a grid at tablet width; T1 as the fallback if the grid is too heavy narrow. Decide during Phase 1 red tests.
- **Discoverability + affordance.** A resting hover/focus tell on editable names (dashed underline or `--grid-minor` wash) so click-to-rename is obvious (today `dim-row__name--head` is an inert uppercase span, `DimensionManager.tsx:171-182`, `base.css:355-359`).
- **Keyboard shortcut.** Bind a global key (proposed `d` = focus the dimension rail's first phantom, same capture-phase + text-field-guard grammar as `c`/`v` at `DesignSurface.tsx:248-311`) so bulk entry starts without the mouse. Must not collide with SITEMAP ¬ß4's global keymap and must no-op while a text field is focused.
- **Tablet (Decision 4).** At <640px `.design-surface-row` already stacks to a column (`base.css:1335-1339`); the rail joins that stack (rail ‚Üí circle ‚Üí register) or becomes a collapsible drawer. The phantom-row inputs are native text inputs (already touch-friendly). No on-ring editing in Phase 1, so tablet risk here is low.

### Phase 2 brief ‚Äî on-ring inline authoring (Direction C, the delight layer)

- **Ghost "+" ring-gap segments.** The layout already leaves a fixed 6¬∞ gap between arcs (`canvasLayout.ts:17` `GAP_RADIANS`). Render a persistent, muted "+" affordance in each gap (and an all-ghost ring when 0‚Äì1 dimensions exist, so the empty state shows the circle's own skeleton and finally answers "what is this circle?"). Clicking a gap inserts a dimension arc at that angular position and opens an **on-ring rename**.
- **On-ring rename via `foreignObject`.** Editing a dimension name (on its arc) or a parameter name (at its dot) mounts a small HTML `<input>` in an SVG `<foreignObject>` positioned from the existing `ArcGeometry.labelPos` / `DotGeometry.labelPos` (`canvasLayout.ts:94-113`). Commit/cancel grammar identical to the rail (Enter/Esc/blur) so the two paths feel like one app. **This is the riskiest surface** (see Open tension) ‚Äî it is deliberately layered on top of a stable Phase 1.
- **Add a parameter on the ring.** A "+" affordance near an arc's outer edge inserts a parameter dot on that dimension; inline-rename at the new dot. Feeds the same `parameters.add(dimensionId, name)` the rail uses.
- **Touch targets (Decision 4).** Every on-ring hit target (gap "+", dot, arc rename handle) uses `dotHitRadiusUnits(measuredWidth)` (`canvasResponsive.ts:21-23`) so it stays ‚â•44px on tablet at any zoom ‚Äî the invisible hit circle Canvas already renders for dots (`Canvas.tsx:277`, `base.css:1149-1151`) is the pattern to extend to arcs and gap-adds.
- **A11y (STYLE_GUIDE ¬ß10).** On-ring editors are focusable, carry `aria-label` ("Rename dimension {name}", "Rename parameter {name}"), and participate in the canvas's roving-tabindex focus order (`Canvas.tsx:342,365`). Ghost "+" segments are `role="button"` with an `aria-label` ("Add dimension"). Focus order stays app-bar ‚Üí context-bar ‚Üí surface ‚Üí composer ‚Üí status (SITEMAP ¬ß2, STYLE_GUIDE ¬ß10.159).

### Throughout both phases ‚Äî visual stability (the north-star clause)

- **Append-only dot placement.** Replace the even-spacing at `canvasLayout.ts:278` (`segmentSpan / (params.length + 1)`) with **fixed increments from the arc start** so dot #k always lands at the same angle regardless of how many params follow, plus **arc-span growth** as an arc fills (so a crowded arc widens rather than re-packing). Existing dots ‚Äî and their bound spokes ‚Äî never move when a parameter is added. This is a pure change to the layout function (ADR-0005: `layout` stays a deterministic `fn(tree)`; no stored x/y), so it is fully unit-testable.
- **Ease any residual dot movement.** Where a dot must move (arc-span growth on a shared arc, or a dimension reorder), reuse the ~120ms reduced-motion-safe transform transition already proven on nodes (`Canvas.tsx:355-362`, STYLE_GUIDE ¬ß8). Dots today only transition `r` (`base.css:1153-1155`); extend the same `transform`/position transition so movement reads as "settling," not "jumping."

### Polish subsection (audit minor findings ‚Äî fold in where cheap, don't gold-plate)

- **Swatch picker is an inline flyout, not a popover** (`DimensionManager.tsx:184`, `base.css:308-314` `flex-basis:100%`) ‚Äî it flex-wraps to a new line and nudges row height, a small reflow. Make it a Radix `Popover` anchored to the swatch, consistent with `RemoveDimensionConfirm` (`DimensionManager.tsx:67-99`).
- **Dimension "Remove" is always-visible full weight** (`DimensionManager.tsx:88-96`); STYLE_GUIDE ¬ß2.2/¬ß6 says per-row destructive actions should be the `rowAction` variant (quiet until row hover/focus). It currently competes with the name on every row.
- **"New context" disabled while composing with no visible reason** (`DesignSurface.tsx:476`) reads as a bug ‚Äî add a `title`/tooltip ("Finish the current draft first").
- **Canvas caps at `max-height: 60vh`** while the register stretches to row height (`base.css:1094` vs `:1298`); a tall register can out-height the circle and weaken the circle-as-hero hierarchy. Consider a canvas min-height floor so the ring stays the visual anchor.

## Open tension (name it explicitly)

**On-ring text editing on tablet is the riskiest part of this issue** ‚Äî hence it is Phase 2, layered on top of a stable Phase 1, not the first thing built. The specific risks: (a) `foreignObject` input positioning must survive SVG scaling and the label-declutter shift (`canvasLayout.ts:192-210`) so the editor lands on its mark at every zoom; (b) touch hit targets for gap-"+", arcs, and dots must not overlap each other or the invisible dot hit circles at small radii on a crowded ring; (c) an on-screen keyboard on tablet reflows the viewport under an SVG-anchored input, which can drift the editor off its target. If Phase 2 proves too costly or fragile on tablet, Phase 1 alone already closes all three user complaints and delivers the fast typed half of the Hybrid ‚Äî Phase 2 is genuinely additive, not load-bearing.
