## Context

089-P7 made the React Flow canvas the DEFAULT workspace (capability-gated ‚â• 1024px + not data-saver); `WorkspaceSurface` is the < 1024px / reduced-data fallback. The e2e strategy that shipped: the 21‚Üí22 `@dev-flag` `d3-canvas.spec.ts` specs are the **canvas** suite; the ~57 non-`@dev-flag` specs were re-pinned to the **WorkspaceSurface fallback** (via `e2e/workspaceSurface.ts::forceWorkspaceSurface`, a `prefers-reduced-data` matchMedia shim). Both surfaces ship, so both are covered ‚Äî but the review flagged that a few behaviors are now tested ONLY on the fallback, plus one plausible latent bug and a touch-device gap. P7 added a canvas `main` landmark + a canvas axe smoke (WCAG2 A/AA serious/critical, design view) which passes; these items go beyond that.

2c. **[SHIPPED 2026-07-20] `Canvas.tsx:130` hit-radius is transform-blind.** **FIXED**, but the fix is NOT the one-liner the report implies ‚Äî two adversarial review rounds reshaped it, and the second BLOCKED the first implementation:
   - **The fix:** `Canvas` gained an optional `scale` prop (default 1 ‚Üí the untransformed `DesignSurface` fallback is unaffected); `DesignRingBody` passes `useStore((s) => quantizeHitScale(s.transform[2]))`. Screen width = `layoutWidth * scale`. Logic lives in pure, unit-tested helpers (`hitRadiusUnits`, `quantizeHitScale`) per the repo pattern. `quantizeHitScale` FLOORS to 0.25 buckets so the ring re-renders only on bucket crossings (~6 per full zoom sweep), never per frame; flooring errs the radius LARGER, never below 44px. The label tier still uses the raw zoom-invariant layout width (item 2 ‚Äî correct, untouched).
   - **Why naive compensation is HARMFUL (review round 1):** `.canvas-dot-hit` is `fill: transparent` = PAINTED, so SVG `visiblePainted` hit-tests it. The circle therefore also swallows background-deselect clicks (`Canvas.tsx`'s `e.target === e.currentTarget`) and drives hover emphasis. Round 1 also refuted its own strongest objection with numbers: at zoom 0.5 the no-overlap cap does NOT bind for realistic rings, so the fix is real, not a no-op. It also caught that a `floor(zoom/0.25)` bucket returns **0** at RF's `minZoom={0.2}` ‚Üí `dotHitRadiusUnits(0)` ‚Üí `Infinity`, silently swallowed by the `Math.min`.
   - **Round 2 BLOCKED the first implementation** (clamping the compensation via `MIN_HIT_SCALE` as the sole containment). That bound is `2 * 22000/W` ‚Äî a function of the LAYOUT WIDTH, and `W` is itself variable: the ring shell is `min(480px, 60vh, 100%)`, so a short viewport shrinks it far below 480 (the ‚â•1024px capability gate constrains WIDTH only). Not a bound at all.
   - **Root cause found + fixed at source:** `maxDotHitRadius` was NOT the containment guarantee it claimed to be. Its per-arc scan is guarded by `if (m >= 2)`, so it only ever measured WITHIN-dimension pairs ‚Äî a ring where no dimension has two dots has real, finite cross-dimension pairs and yet fell through to an effectively open `ARC_RADIUS`. Harmless while the radius was small; **not** once it is screen-space-sized, because in compose mode an overlapping circle carries the bind handler ‚Üí **the WRONG parameter binds** (a data-correctness bug, and a re-admission of the 082/5bbc8bc overlap regression). `maxDotHitRadius` is now a TRUE all-pairs minimum ‚Äî which is what its own comment always CLAIMED. Identical value whenever some dimension has m ‚â• 2, so the normal case is unchanged. NB the old within-dimension argument was SOUND but incomplete: cross-gap pairs genuinely are wider, but that only bounds the pairs the loop visits. `MIN_HIT_SCALE` is retained ONLY for the divide-by-zero guarantee; containment is `maxDotHitRadius`'s job. A pre-existing test asserting the old behavior ("when no dimension has two dots to overlap") had a FALSE premise ‚Äî there are two dots, in different dimensions ‚Äî and was rewritten to the corrected contract.
   - Also corrected: a factually wrong rationale comment claiming the LOD had collapsed editable surfaces at that zoom (the ring has NO LOD at any zoom; `LOD_ZOOM` is the register's and is disabled during compose), and three tests that asserted less than their names claimed (one recomputed its expectation from the implementation, so it would have passed at any `MIN_HIT_SCALE`).
   Verified: 1660 unit tests, full e2e 84/84, tsc + eslint clean. Original report below.

   **[FOUND] `Canvas.tsx:130` hit-radius is transform-blind (touch-target shrinks under zoom).** `dotHitRadiusUnits(width)` is fed the zoom-invariant `contentRect` width but its own comments require the ON-SCREEN width so the dot hit target stays ~44px "at any zoom". At zoom 0.5 the physical target is ~22px ‚Äî a real WCAG 2.5.5 target-size shrink under canvas zoom (the exact INVERSE of the label-tier non-issue: this consumer WANTS screen-space and gets layout-space). **Fix:** feed a screen-space width (e.g. track the live RF zoom and multiply, or read `getBoundingClientRect().width`) into the hit-radius calc only. Pairs with the touch/tablet item (#4).

4. **Touch / tablet verification ‚Äî emulated-touch specs AUTHORED + PROVEN-PASSING but HELD (`test.fixme`); pointer-coarse nudge DECLINED; real-device pass is the irreducible manual remainder.**
   - **Authored (all passed in CI once), in `d3-canvas.spec.ts`, `@dev-flag`, now `test.fixme`:** ‚ëÝ a single-finger touch-drag on the empty pane **pans** the viewport; ‚ë° a touch-drag on an Architecture table-node **header reorders + persists `sort`** (exact touch twin of the mouse drag-reorder spec); ‚ë¢ a **touch tap focuses** the Foundation phantom (tap-to-activate). Drags use CDP `Input.dispatchTouchEvent` (the high-fidelity path React Flow's d3-zoom listens to; Playwright's `page.touchscreen` only exposes `tap`); the tap reuses `page.touchscreen.tap` in a `hasTouch` context. **Why held:** all three passed, but each opens a fresh `hasTouch` browser context (two also open CDP sessions), which raised peak full-suite e2e load enough to tip the documented, load-sensitive **100-D child-core mount-timing flake** (`drilling Œ± promotes a LIVE child core`, Œ±1 `toBeVisible`) into losing all 3 retries **two consecutive runs** ‚Äî it had been green all day until these landed. `test.fixme` banks the proven CDP-touch approach with zero runtime load; **re-enable in a dedicated/serial touch e2e lane** (or once the 100-D spec's mount-timing is hardened) so they don't contend with the canvas specs.
   - **`pointer: coarse` / `hover: none` gate nudge ‚Äî INVESTIGATED, DECLINED (do NOT change the gate).** `canvasCapable()` (`src/store/canvasMode.ts:40`) gates on **width ‚â• 1024px + not `prefers-reduced-data`** only ‚Äî no pointer-media query. Adding a `pointer: coarse` + narrow nudge to the WorkspaceSurface fallback would **REVERSE the deliberate 089 tablet-first design**: iPad-landscape (~1024 CSS px, `pointer: coarse`, `hover: none`) is exactly the device class the canvas is designed FOR, and would be the class such a nudge strips it from. Small phones (the only genuine coarse-pointer concern) are already excluded by the < 1024px width gate. `hover: none` alone does not justify a fallback ‚Äî hover-driven emphasis (item 1) is a progressive enhancement that degrades gracefully to tap-select on touch. Changing a product-visible surface-selection gate demands real-device validation that no agent can perform; the emulated-touch specs above already show the canvas is touch-operable. **Recommendation: keep the gate as-is.** Revisit ONLY if the manual real-device pass below surfaces a concrete unusability (e.g. pinch-zoom or node-drag genuinely broken on a real iPad) ‚Äî that data, not a speculative gate change, should drive any nudge.
   - **IRREDUCIBLE MANUAL REMAINDER (no automation can close):** a real-device pass on an actual touch tablet for **multi-touch pinch-to-zoom** (needs two tracked contact points + platform gesture recognition ‚Äî not faithfully emulable) and the tactile fidelity of touch pan/node-drag (momentum, contact slop, palm rejection). Emulated single-finger touch is covered; two-finger pinch is not.

5. **[SHIPPED 2026-07-22, `6434752`] Harden the cross-node Tab spec against full-suite flake.** It passed 3/3 in isolation but flaked once under full `npm run e2e` parallelism (focus-timing across table nodes via the `onExitBoundary` seam). The two retry-mitigated `toBeFocused()` asserts (`d3-canvas.spec.ts`, forward + backward boundary) are now deterministic `expect.poll(activeElement)` checks of the SAME crossed-node + landed-cell invariant, so the spec no longer relies on `retries: CI ? 2 : 0` to survive the rAF/focus-pan race (the HANDOFF e2e lesson: poll `activeElement`, never snapshot it once).
