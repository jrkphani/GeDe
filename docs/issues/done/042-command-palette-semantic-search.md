# 042: Command palette — semantic search (client-side embeddings)

- **Status**: SHIPPED (integrated into the v2 wave, PR #7 — see Shipped notes)
- **Milestone**: M6 (UX) / M2 (palette)
- **Blocked by**: 017 (the command palette + ranking seam — shipped)

## Slice

As a user, ⌘K finds commands and contexts **by meaning, not just by matching letters** — typing "hide the unconnected" surfaces the adjacency-emphasis toggle, "money" surfaces a context named "budget" — while the current fast lexical ranking still wins for exact/prefix matches. All of it runs **in the browser**: no server call, no data leaves the device, works offline, **zero AWS run cost**.

## Motivation

The palette (017) ranks purely lexically — `src/domain/paletteRanking.ts` scores symbol → name → justification, recents float. That's great for "I know the name" but misses "I know what I mean." Semantic search closes that gap. Doing it **client-side with embeddings** keeps it consistent with the whole local-first bet (ADR-0003: PGlite in the browser, data on-device): it costs **nothing** to run on AWS, needs no network, and never sends the user's domain content anywhere — the opposite of a hosted embeddings API. The only real cost is a one-time model download, cached per device.

## Scope

- **In-browser embeddings** via `transformers.js` (a small sentence-embedding model, e.g. quantized MiniLM ~20MB). Loaded **lazily** (first palette open, or an idle prefetch) and cached by the service worker so it's offline-capable and downloaded once.
- **Item corpus**: the palette's searchable set — commands from `useCommandRegistryStore` (`collect()`), plus live contexts (name + justification) already exposed to the palette (017's `coreCommands`). Embed each item's text once; **cache vectors in IndexedDB keyed by a content hash** so re-embedding only happens when an item's text changes.
- **Query path**: on keystroke (debounced), embed the query, cosine-similarity against the cached item vectors, and **blend the semantic score into `paletteRanking.ts`** (a weighted combination with the existing lexical/recency signal) — one pure, tested ranking function, not a parallel search.
- **Graceful degradation**: until the model is loaded (or if it fails / the device can't), the palette works **exactly as today** (pure lexical). Semantic is strictly additive — it never blocks input or the first result.

Out of scope: server-side/hosted embeddings (explicitly rejected on cost + privacy — see Design brief); semantic search *outside* the palette (a separate future issue if wanted); RAG/generation of any kind.

## Design brief

- **Local-first, $0 to run** (ADR-0003, TECH_STACK §5): embeddings run on-device like PGlite does; no Bedrock/OpenAI/pgvector, no per-query cost, no data egress. This is the whole reason to do it client-side.
- **Fast path stays fast**: exact/prefix lexical matches must not regress — semantic is a *blend*, tuned so a perfect name match still ranks first; the query embed is debounced and never gates the visible results.
- **Progressive, never a wall**: the palette is fully usable before/without the model. Model load is invisible (lazy + SW-cached), and a failure degrades silently to lexical.
- **Honest about the one cost**: a ~20MB model download per device (cached, offline after). Load it on demand, not in the initial bundle — the app's first paint and PGlite boot must not pay for it.

**References**: issue **017** (command palette, `commandRegistry`, `paletteRanking.ts`, `CommandPalette.tsx`) · **ADR-0003** (Postgres/PGlite local-first, the on-device precedent) · ADR-0005 (determinism — cache vectors, don't recompute nondeterministically) · TECH_STACK §5 (zero-friction PWA), §6.2 (SW caching — the model is a cacheable asset) · STYLE_GUIDE §10 (palette keyboard/focus) · SITEMAP (⌘K).

## Test-first plan

1. **Pure blend ranking**: a unit test of the extended `paletteRanking` — given lexical scores + semantic similarities, a perfect name match still ranks first, and a semantically-close/lexically-far item outranks an unrelated one. No model needed (feed synthetic vectors).
2. **Graceful fallback**: with the embedding model absent/unloaded, the palette returns the exact current lexical results (a golden test that semantic is additive-only).
3. **Vector cache invalidation**: changing an item's text re-embeds it; unchanged items reuse the cached vector (content-hash keyed) — asserted without a real model via a stubbed embedder.
4. **No network / no cost**: a test/guard that the embedding path makes **no fetch to any AWS/remote endpoint** (the model loads from the SW cache/bundled asset only).

## Acceptance criteria

- [x] ⌘K surfaces semantically-relevant commands/contexts (meaning, not just letters), blended with — not replacing — the lexical/recency ranking; exact matches still rank first. (`rankCommands`'s tiered blend in `src/domain/paletteRanking.ts`; property-tested "exact match never loses" invariant.)
- [x] Fully functional (lexical) before the model loads and if it fails; model is lazy-loaded, **not** in the initial bundle. *(SW-cached is a partial exception — see note below: this repo has no registered service worker yet, so offline-after-first-load rides on the browser's Cache Storage API directly, which `transformers.js` uses internally regardless of an app-level SW.)*
- [x] Vectors cached in IndexedDB, re-embedded only on content change. (`src/lib/vectorCache.ts`'s `createIndexedDbVectorCache` + content-hash-keyed `embedWithCache`; falls back to an in-memory cache when IndexedDB is unavailable — same-session-only degradation, never a crash.)
- [x] **Zero AWS run cost and zero data egress of user content** — the embedding computation runs entirely on-device (WASM/onnxruntime-web); no Bedrock/OpenAI/hosted-embedding call, ever. *(One caveat worth a deliberate read: the model **weights** are fetched once from Hugging Face's CDN, not AWS and not vendored same-origin — see Shipped notes.)*
- [x] `npm run verify` green (462 unit/component + 43 e2e); bundle-size impact of the model is out-of-initial-chunk (measured — see Shipped notes).

## Shipped notes (2026-07-07)

- **Pure blend** (`src/domain/paletteRanking.ts`): `rankCommands` now takes an options bag (`{ limit?, semanticScores? }`, backward compatible — every 017 call site and test is untouched). A lexical match's tier (0-4, unchanged from 017) always outranks a semantic-only hit (tier 5, gated by `SEMANTIC_INCLUDE_THRESHOLD = 0.35`); within a shared tier, semantic similarity is a secondary sort key ahead of recency. `cosineSimilarity`/`semanticScoresFromVectors` are pure, hand-vector-testable — no model needed for any of `paletteRanking.test.ts`'s 18 tests (includes a `fast-check` property test for the "exact match never loses" invariant).
- **Vector cache** (`src/lib/vectorCache.ts` + `src/domain/contentHash.ts`): content-hash keyed (FNV-1a-style, non-cryptographic — it's a cache key, not a security boundary), `embedWithCache(text, embedder, cache)` re-embeds only on a content change, dedupes in-flight races for the same key. `createIndexedDbVectorCache` is the real persistent store; `createMemoryVectorCache` is both the tested fallback (no IndexedDB in Node/jsdom) and the production degrade path when IndexedDB is unavailable.
- **Embedder** (`src/lib/semanticEmbedder.ts`): **`@huggingface/transformers` v4** (the actively-maintained successor to the issue's suggested `@xenova/transformers` — same model format/namespace, but the old package's onnxruntime-web/protobufjs chain carries 4 high + 1 critical advisories that the new package has fixed; verified via `npm audit` that this repo's vulnerability count is unchanged from before this dependency — 5, 4 moderate + 1 high, all pre-existing in drizzle-kit/drizzle-orm, unrelated). Model: `onnx-community/all-MiniLM-L6-v2-ONNX`, `dtype: 'q8'` (quantized), `pooling: 'mean', normalize: true`. Dynamically imported only inside `embed()`, never at module load.
- **Orchestration** (`src/store/semanticSearch.ts`): a small Zustand store — `ensureModel()` (idempotent lazy load, `idle → loading → ready|unavailable`) and `scoreQuery(query, items)` (embeds the query + every item's `itemContentText`, both through the cache, then writes `{query, scores}`; a generation counter discards a stale in-flight result superseded by a newer keystroke — the HANDOFF generation-counter pattern, applied here for the same reason it was needed elsewhere: out-of-order async completion). Both are fully unit-tested against **injected stub factories** (`__setSemanticSearchFactoriesForTest`) — this suite never imports the real model or touches the network, including an explicit guard test that `scoreQuery` never calls `fetch` while the model is unloaded.
- **Wiring**: `AppShell.tsx` calls `ensureModel()` on the palette's first open (both the ⌘K shortcut and the toolbar button) — **not** `CommandPalette.tsx` itself, so the palette component stays feature-agnostic (017's framing) and, critically, so `CommandPalette.test.tsx` never triggers the real model/network just by rendering the palette. `CommandPalette.tsx` debounces (150ms) a `scoreQuery` call on the live query and blends `useSemanticSearchStore`'s `scores` into `rankCommands` only when the store's `query` matches the current (trimmed) query — a stale/in-flight result never gets applied to the wrong keystroke.
- **Bundle size (measured)**: `npm run build` puts `@huggingface/transformers` + onnxruntime-web in their own lazy chunk (`transformers.web-*.js`, 549.14 kB / gzip 158.92 kB) — confirmed via the Workbox precache manifest it's also a real, separate SW-precachable asset, and it never lands in the app's main entry chunk. A same-repo-checkout baseline build off `origin/main` (before this branch) measured the main chunk at 1,209.26 kB / gzip 316.23 kB; this branch's main chunk is 1,213.40 kB / gzip 317.67 kB — a **+4.14 kB / +1.44 kB gzip** delta (the store/lib orchestration code only), confirming the model library itself is fully out-of-initial-bundle.
- **Known gap — SW registration doesn't exist yet in this repo** (`vite.config.ts`: `injectRegister: false`, "no SW registration until the update-UX from TECH_STACK §6.2 lands with the shell" — predates 042, listed in HANDOFF's deferred threads). So "SW-cached" is not literally true today: there is no app-level service worker controlling anything. In practice, `transformers.js` uses the browser's Cache Storage API directly (`env.useBrowserCache = true`, set explicitly in `semanticEmbedder.ts`) — Cache Storage is available to any page regardless of whether a service worker is registered, so the model **is** cached and **is** available offline after first load; it just isn't precached by *our* SW ahead of time the way `docs/TECH_STACK.md` §6.2 envisions. Once SW registration lands, `transformers.web-*.js` is already in the generated Workbox precache manifest (confirmed above) — no 042 code change will be needed then, just turning `injectRegister` on.
- **A real, larger-than-the-issue-implied one-time cost**: confirmed live (via a Playwright network probe against the real dev server) that opening the palette and typing a query fetches, from `huggingface.co`: `config.json`, `tokenizer.json`/`tokenizer_config.json`, and `onnx/model_quantized.onnx` — plus onnxruntime-web's own WASM runtime (`ort-wasm-simd-threaded.asyncify-*.wasm`, ~23.5MB, separately excluded from SW precache exactly like PGlite's WASM). The issue's design brief frames the one-time cost as "~20-23MB model download" — in practice it's that **plus** the onnxruntime-web WASM runtime, so closer to **~45MB total** the first time semantic search actually runs. Still $0 AWS, still fully cached/offline after, still zero egress of any user content — but a bigger number than the issue text implies, worth knowing.
- **Decision needing review — e2e now has a real, live third-party network dependency.** `command-palette.spec.ts` (and any future e2e that opens the palette and types a non-empty query) now triggers a real fetch to `huggingface.co` once the 150ms debounce fires, because `AppShell.openPalette()` calls the real `ensureModel()`. This ran clean in this session (all 43 e2e green, 39.9s total, no flakes, no hang — confirmed network access from this environment reaches Hugging Face fine), but it is a **new category of external dependency at the `verify` gate** that didn't exist before (every other e2e dependency is fully local/offline — PGlite's WASM is bundled, fonts are self-hosted). If CI ever runs with restricted egress, or Hugging Face has an outage, this is a plausible new flake source distinct from the documented cold-start ones. Flagging for a decision rather than silently working around it (e.g., an e2e-only flag to skip `ensureModel()` would diverge test behavior from production, which seemed like the wrong trade to make unilaterally).
- **Model choice**: `onnx-community/all-MiniLM-L6-v2-ONNX` (the maintainer-recommended id for this package version, superseding the issue-suggested `Xenova/all-MiniLM-L6-v2` id — same underlying model). Not swapped for a smaller variant (e.g. a `bge-micro`) — the issue's own item-count assumption (dozens-hundreds of commands/contexts) means this was never a latency-bound choice; the corpus-side embed cost is dominated by network/model-load, not the similarity scan, exactly as the issue's implementation notes anticipated.
- **No migration** (client-only, as scoped).

## Implementation notes

- **Model**: quantized `all-MiniLM-L6-v2` (~23MB) is the safe default; a smaller model (e.g. a `bge-micro`/6-layer variant) trades a little quality for a smaller download — pick during implementation against the item-count/latency budget. Command/context corpora are small (dozens–hundreds), so query latency is dominated by the single query-embed, not the similarity scan.
- **Loading**: `transformers.js` with the model served as a same-origin cached asset (SW `registerType: 'prompt'`, §6.2) so it's downloaded once and available offline; lazy-import so it's a separate chunk (Vite `optimizeDeps`/dynamic import — watch the PGlite exclude precedent).
- **Blend weighting**: start with lexical-dominant (perfect-match wins), semantic as a tiebreaker/recall booster; expose the weight as a constant and property-test the "exact match never loses" invariant.
- **Determinism**: embeddings are deterministic for a given model+input; cache by content hash and never recompute in a way that reorders results within a session (ADR-0005 spirit).
- **Scope creep guard**: this is palette-only. If semantic search over the whole corpus (coverage matrix, register) is wanted later, it's a separate issue that can reuse this embedder + vector cache.
